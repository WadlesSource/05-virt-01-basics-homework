Задача 2
1. Высоконагруженная база данных MySQL, критичная к отказу
Рекомендуемый вариант: Физические сервера
ПДля таких баз данных самое важное – чтобы диски работали очень быстро, а ещё чтобы не было никаких задержек, которые может создавать гипервизор. На физическом сервере ты получаешь прямой доступ ко всему оборудованию – NVMe дискам, процессору, оперативной памяти.
Надежность обеспечивается либо через само приложение (например, Master-Slave репликация или Galera Cluster), либо если используются два физических сервера с общим хранилищем данных.
Если денег не так много, можно попробовать паравиртуализацию (KVM/VMware), но тогда нужно обязательно зарезервировать оперативку и процессор, чтобы избежать ситуации, когда другие программы на той же машине замедляют работу базы.
2. Различные web-приложения
Виртуализация уровня ОС (Контейнеризация, Docker/Kubernetes)
Веб-приложениям обычно нужна гибкость, их нужно быстро увеличивать или уменьшать, и быстро запускать. Контейнеры совсем легкие, запускаются моментально и позволяют уместить много разных сервисов на одном компьютере.
Это идеальный вариант для процессов постоянной интеграции и доставки (CI/CD).
Что ещё можно: Можно использовать и обычные виртуальные машины, если приложениям нужны разные версии операционных систем или им обязательно нужно свое отдельное ядро.
3. Windows-системы для использования бухгалтерским отделом
Паравиртуализация (Гипервизоры: VMware, Hyper-V, KVM)
Бухгалтерские программы (такие как 1С, клиент-банки) обычно работают только на Windows. Виртуальная машина очень удобна тем, что можно легко сделать резервную копию всей системы, быстро ее восстановить, настроить, кто к чему имеет доступ, и сымитировать нужную сеть.
Отлично подходит для создания стандартных рабочих мест (VDI), а также легко добавлять или уменьшать ресурсы, например, оперативную память или процессор.
4. Системы, выполняющие высокопроизводительные расчёты на GPU
Физические сервера
Чтобы получить максимум от вычислений (будь то искусственный интеллект, машинное обучение или рендеринг), нужен прямой доступ к видеокарте без каких-либо посредников.
Можно попробовать паравиртуализацию с прямой передачей GPU (vGPU). Это пригодится, если нужно разделить одну очень мощную карту (вроде NVIDIA A100/H100) между несколькими задачами. Всё же, для большинства действительно важных расчетов, лучше использовать обычный физический сервер.
Задача 3
Сценарии:
1. 100 виртуальных машин на базе Linux и Windows, общие задачи, нет особых требований. Преимущественно Windows based-инфраструктура, требуется реализация программных балансировщиков нагрузки, репликации данных и автоматизированного механизма создания резервных копий.
VMware vSphere (ESXi + vCenter)
Для инфраструктуры в 100 машин с требованиями к автоматизации (vCenter Server), репликации (vSphere Replication) и высокой надежности, VMware является стандартом индустрии. Программный балансировщик (например, HAProxy) легко внедряется внутри ВМ, а vCenter обеспечит централизованное управление и создание резервных копий через сторонние API (Veeam).
2. Требуется наиболее производительное бесплатное open source-решение для виртуализации небольшой (20-30 серверов) инфраструктуры на базе Linux и Windows виртуальных машин.
Proxmox VE
Это наиболее сбалансированное решение, сочетающее KVM для высокой производительности и LXC для контейнеров. Proxmox полностью бесплатен (включая все функции кластеризации), обеспечивает отличную работу как Linux, так и Windows ВМ, а также имеет встроенные средства резервного копирования.
3. Необходимо бесплатное, максимально совместимое и производительное решение для виртуализации Windows-инфраструктуры.
Microsoft Hyper-V Server (standalone или Windows Server с ролью Hyper-V)
Если принципиальна максимальная совместимость с Windows-инфраструктурой, Hyper-V — лучший выбор. Он обеспечивает нативную производительность для гостевых систем Windows, отличное управление через PowerShell и Windows Admin Center, а также использует стандартные драйверы интеграции.
4. Необходимо рабочее окружение для тестирования программного продукта на нескольких дистрибутивах Linux.
KVM + libvirt (или KVM/QEMU)
Для тестового окружения KVM обеспечивает максимальную производительность, так как является частью ядра Linux. Он легко управляется через virsh или GUI-инструменты (Virt-manager), позволяя быстро создавать, клонировать и удалять ВМ с разными дистрибутивами Linux.
Задача 4
Опишите возможные проблемы и недостатки гетерогенной среды виртуализации (использования нескольких систем управления виртуализацией одновременно) и что необходимо сделать для минимизации этих рисков и проблем. Если бы у вас был выбор, создавали бы вы гетерогенную среду или нет?
Проблемы и недостатки гетерогенной среды:
Несколько консолей управления: Администраторам приходится переключаться между разными системами (vCenter, SCVMM, Proxmox и т.д.), что удлиняет время реакции на инциденты.
Разные навыки: Требуется персонал, владеющий знаниями по нескольким гипервизорам, что сложно и дорого.
Различия в защите: Разные платформы имеют уникальные уязвимости; защита одной (например, VMware) не покрывает другую (KVM), что усложняет аудит безопасности.
Несовместимость и интеграция: Миграция виртуальных машин (Live Migration) между разнородными средами затруднена или невозможна без простоев.
Мониторинг: Разные системы выдают метрики в разных форматах, что требует сложной настройки систем мониторинга для общей картины.
Увеличение операционных расходов (OPEX): Лицензирование: Необходимость оплаты поддержки нескольких вендоров.
Ошибки администраторов: Высокий риск человеческой ошибки из-за путаницы между системами.
Что необходимо сделать для минимизации этих рисков и проблем?
Централизация управления: Использование сторонних решений (например, ManageIQ или HashiCorp Terraform) для унификации управления разнородными гипервизорами.
Стандартизация процессов: Внедрение единых процедур резервного копирования, резервного восстановления и управления патчами для всех систем.Автоматизация: Применение Ansible, Puppet или Chef для конфигурации, что исключает ошибки ручного управления в разных консолях.
Единый мониторинг: Настройка Zabbix или Prometheus для сбора метрик со всех платформ.
Создавать ли гетерогенную среду? Нет, по возможности следует избегать создания гетерогенной среды.
Лучшая стратегия — это гомогенная среда (один вендор). Это обеспечивает максимальную совместимость, упрощает администрирование, снижает затраты на обучение и обеспечивает единую политику безопасности. Гетерогенную среду стоит рассматривать только в исключительных случаях, таких как: Слияние двух компаний с разной инфраструктурой. Строгие требования к облачной независимости (multi-cloud strategy). Необходимость специфических функций, которые поддерживает только определенный гипервизор.
